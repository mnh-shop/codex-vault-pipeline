## 6. Retrieval and Semantic Search Breakdown

### Exact/Path Lookup
- **SQLite** queries on `packs`, `pack_files`, `pack_chunks` tables
- Source ID filtering: `WHERE source_id = ?`
- Path filtering: `WHERE path = ?`

### SQLite/FTS/BM25
- **FTS5 virtual table:** `pack_chunks_fts` over chunk text
- Query processing: split on whitespace, escape FTS5 chars, join with OR
- Ranking: FTS5 built-in BM25 rank
- Snippet generation: `snippet(pack_chunks_fts, 4, '<b>', '</b>', '...', 64)`

### LanceDB/Vector Search
- **Legacy only:** `build_indexes.py` creates LanceDB with `all-MiniLM-L6-v2` embeddings
- **v2:** NOT IMPLEMENTED — no v2 vector index exists
- Legacy vector search available via `retrieval.py search --mode vector`

### Semantic Search
- **Where embeddings created:** `build_indexes.py` → `SentenceTransformer("all-MiniLM-L6-v2")` → 384 dims
- **Which files embedded:** Units from `.runtime/units/**/*.json` (markdown sections, code symbols, workflow nodes)
- **Raw files embedded:** NO — only extracted units
- **60-sources notes embedded:** NO — only .runtime/units/
- **20-domains/30-concepts/40-use-cases/50-guides embedded:** NO
- **Generated catalogs embedded:** NO (by policy: `generated_catalog_low_priority`)
- **READMEs embedded:** NO (by policy: `readme_low_priority`)
- **Code embedded:** YES — as units extracted from code files
- **JSON workflows embedded:** YES — as units extracted from workflow JSON
- **n8n workflows as domain records:** YES — `.runtime/domain/n8n-workflows/` contains workflow metadata
- **Oversized repos excluded:** Repomix excludes files >1MB, repos >100MB
- **README/catalog dominance controlled:** Demotion penalty (-0.4 for README, -0.5 for catalogs) in context packer scoring
- **Provenance in results:** YES — source_id, path, artifact_id, chunk_id preserved in pack_chunks table
- **Current indexes stale:** YES — built before vault restructure, but still valid (reads from raw/, not wiki/)

### Source Routing
- **5 rules** in `context_packer.py`:
  1. AgentField: `agentfield|agent.definition|protocol` → boost +0.5
  2. n8n docs: `n8n.*error.handling|n8n.*credentials` → boost +0.5
  3. OSINT: `osint|reconnaissance|intelligence.gathering` → boost +0.4
  4. Memory: `memory.system|context.persistence|mnemosyne` → boost +0.5
  5. Coding agents: `coding.agent|autonomous.code|code.generation` → boost +0.3

### Context Pack Selection
- **Algorithm:** Greedy by score descending
- **Constraints:** max_tokens (8000), max_items (50), per_source_limit (15), per_file_limit (5)
- **Demotion:** README (-0.4 unless query matches readme terms), catalog (-0.5 unless query matches catalog terms)

### Hybrid/Fusion Logic
- **Not implemented in v2.** Retrieval policy declares hybrid as candidate (Haystack pattern).
- **Legacy:** `retrieval.py` has `hybrid_search()` combining FTS + vector results.

## 11. n8n Scanner Audit

### Credential Semantics Bug (BLOCKER)

**Current behavior:**
- `_detect_credential_security()` scans raw JSON text for patterns: `credentials`, `password`, `secret`, `api_key`, `webhook_url`, `bearer`, `authorization`
- ANY match sets `credential_security_flag = true`
- **Result:** 2,913 workflows flagged — almost all false positives

**The problem:** Every n8n workflow that uses credentials has a `"credentials"` field in its node configuration. This is NORMAL n8n behavior:
```json
{
  "type": "n8n-nodes-base.slack",
  "credentials": {
    "slackApi": {
      "id": "123",
      "name": "My Slack Account"
    }
  }
}
```
This contains `"credentials"` and triggers the security flag — but it's just a reference, not a leaked secret.

### Required Distinction
| Field | Meaning | Current | Correct |
|-------|---------|---------|---------|
| `credential_type_present` | Node references a credential type | Part of `has_credentials` | Separate field |
| `credential_reference_present` | Normal `"credentials": {"id": ..., "name": ...}` | Conflated with security flag | Should NOT flag security |
| `credential_value_present` | Actual values like `sk-...`, `ghp_...` | Not detected | Should flag |
| `secret_value_detected` | High-entropy strings in credential positions | Not detected | Should flag |
| `security_state` | Overall security classification | `credential_security_flag` (bool) | `clean` \| `credential_refs_only` \| `potential_secret_leak` |

### Other Audit Findings
- **Counts scoped to acquired files:** YES (correct)
- **Partial corpus coverage:** Not tracked (should add expected totals)
- **Invalid JSON handling:** Safe (classified as `invalid`, logged in validation_report.json)
- **Metadata classification:** Mostly reliable (metada-*.json detection works, fallback to user_name keys is fragile)
- **Duplicate detection:** Content-only (SHA-256 of raw bytes, no normalization)
- **Connection counting:** Not applicable (scanner counts nodes, not connections)
- **Output schema:** Matches 05E1 design (22 fields)
- **Test coverage gaps:** No tests for credential reference vs value distinction
